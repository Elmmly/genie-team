# Elixir Stack Profile

Stack profile template for Elixir projects (Phoenix, LiveView, Ecto, OTP). Used by `/arch:init` to generate project-specific configuration.

## Detection

| Indicator | Source |
|-----------|--------|
| `mix.exs` | Primary indicator |
| `.tool-versions` → `elixir` | Version source (asdf) |
| `elixir --version` | Version source (system) |

```

## Rules Content

Generate as `.claude/rules/stack-elixir.md`:

```markdown
# Elixir Stack Rules

## Version: Elixir {version}

## OTP Conventions
- **Supervision trees:** Every long-running process MUST be supervised. Use `Supervisor.child_spec/1` or module-based supervisors
- **GenServer callbacks:** Implement ALL callbacks you handle — `handle_call/3` for sync, `handle_cast/2` for async fire-and-forget, `handle_info/2` for messages and monitor signals
- **Always add a catch-all `handle_info/2`:** `def handle_info(_msg, state), do: {:noreply, state}` — prevents crashes from unexpected messages
- **Restart strategies:**
  - `:one_for_one` — restart only the crashed child (default, most common)
  - `:one_for_all` — restart all children (tightly coupled processes)
  - `:rest_for_one` — restart crashed child and those started after it
- **Process linking:** Use `Process.monitor/1` (unidirectional) over `Process.link/1` (bidirectional) unless crash propagation is intended
- **State initialization:** Use `handle_continue/2` for expensive init work instead of blocking `init/1`
- **Naming:** Register processes via `{:via, Registry, {MyApp.Registry, key}}` — not global atoms

## Modern Patterns
- Use `with` chains for multi-step operations that may fail:
  ```elixir
  with {:ok, user} <- fetch_user(id),
       {:ok, token} <- generate_token(user) do
    {:ok, %{user: user, token: token}}
  end
  ```
- Use the pipeline operator `|>` for data transformations — keep functions in pipelines side-effect-free
- Use `@behaviour` for polymorphism — define callbacks, implement in modules
- Use pattern matching in function heads instead of conditionals:
  ```elixir
  def process(%{status: :active} = user), do: activate(user)
  def process(%{status: :inactive} = user), do: deactivate(user)
  ```
- Use structs with `@enforce_keys` for domain models

## Error Handling
- Use tagged tuples `{:ok, value}` / `{:error, reason}` at API boundaries
- NEVER use `try/catch` for flow control — that is for truly exceptional/unexpected situations
- "Let it crash" applies to OTP processes under supervision — NOT to request handling or user-facing operations
- At boundaries (controllers, API endpoints): match on `{:ok, _}` / `{:error, _}` and return appropriate responses
- Use `with` + `else` for multi-step error handling:
  ```elixir
  with {:ok, user} <- fetch_user(id),
       {:ok, _} <- authorize(user, :admin) do
    {:ok, user}
  else
    {:error, :not_found} -> {:error, :user_not_found}
    {:error, :unauthorized} -> {:error, :forbidden}
  end
  ```

## Ecto
- Use changesets for ALL data validation — never validate in controllers
- Compose queries with named functions:
  ```elixir
  User
  |> where_active()
  |> where_role(:admin)
  |> order_by_name()
  |> Repo.all()
  ```
- Use `Ecto.Multi` for transactions with multiple operations:
  ```elixir
  Multi.new()
  |> Multi.insert(:user, user_changeset)
  |> Multi.insert(:profile, fn %{user: user} -> profile_changeset(user) end)
  |> Repo.transaction()
  ```
- Always use `Repo.insert/2` with changesets, not raw maps
- Define `changeset/2` and specific changesets (`registration_changeset/2`) on schema modules

## Anti-Patterns
- **No macro authorship** unless building a library — macros are hard to debug and poorly generated by AI
- **Verify `Enum` functions against hexdocs** — Claude hallucinates non-existent `Enum` functions. When uncertain, verify at hexdocs.pm/elixir
- **No `String.to_atom/1` with user input** — atoms are not garbage collected; use `String.to_existing_atom/1` or keep as strings
- **No hardcoded config values** — use `Application.compile_env(:app, :key)` or runtime config
- **No blocking operations in GenServer callbacks** — offload to `Task.Supervisor` and handle result in `handle_info/2`
- **No `Process.sleep/1` in production code** — use `Process.send_after/3` or `:timer.send_interval/2`

## Verification
After editing Elixir files, run: `mix compile --warnings-as-errors && mix format --check-formatted`
```

## Known Pitfalls

1. **OTP supervision incorrect by default**: Claude generates unsupervised processes, missing `handle_info` catch-alls, and incorrect restart strategies. The OTP Conventions rules section is the primary mitigation.
2. **Macro metaprogramming**: Claude writes macros when functions suffice. Macros are compile-time, harder to debug, and AI-generated macros are frequently incorrect. Rules explicitly discourage macro authorship.
3. **Standard library hallucinations**: Claude invents non-existent `Enum`, `Map`, and `String` functions. The "verify against hexdocs" rule catches this.
4. **Ash Framework DSL**: Ash uses a DSL-heavy approach poorly represented in training data. If detected, add `usage_rules` with current API examples from ash-hq.org. Flag as high-risk in framework detection.
5. **Niche Hex library APIs**: Claude hallucinates APIs for less-popular Hex packages. Always verify function signatures against hexdocs.pm for packages outside the standard library.
6. **"Let it crash" misapplication**: Claude applies OTP's "let it crash" philosophy to request-handling code, omitting error handling at API boundaries. Rules clarify: "let it crash" is for supervised OTP processes, NOT for controllers or user-facing operations.
